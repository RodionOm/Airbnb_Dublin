# Dublin Airbnb — Market Analysis & Price Prediction

> End-to-end data science project on Dublin's short-term rental market: from
> raw 79-column scrape to deployable price model with SHAP explainability.

**Stack:** Python · pandas · scikit-learn · XGBoost · SHAP · VADER · Folium · Power BI
**Dataset:** [Inside Airbnb — Dublin, 16 Sep 2025](https://insideairbnb.com/dublin/) — 6,945 listings, 270k reviews, 2.5M calendar rows
**Author:** Rodion Omelich · [GitHub](https://github.com/RodionOm)

---

## 🎯 Business questions

1. **What drives nightly price in Dublin?** Can we predict it from listing attributes?
2. **Which neighbourhoods are most profitable?** What's the supply structure?
3. **How concentrated is the market?** What share belongs to commercial multi-listing hosts?
4. **What do guests actually complain about?** Sentiment patterns from review text.

## 🔬 Methodology highlights

### Cleaning
- **Price** parsed from `"$1,234.00"` strings; ~25% missing in raw data (cannot impute, dropped for modeling only)
- **Booleans** `t`/`f` → nullable `bool`; **percentages** `"95%"` → 0.95
- **IQR trim** at 1–99% percentile instead of arbitrary `price <= 2000` cutoff
- **`log1p(price)` as target** — handles right-skew, residuals interpretable as % error

### Feature engineering — 30+ derived features
- **Distance to 6 Dublin anchors** (city centre, Temple Bar, Trinity, Guinness, airport, IFSC) via geodesic
- **16 amenity flags** parsed from JSON-like string + total amenities count
- **Host features:** tenure in years, commercial-host flag (5+ listings)
- **Text signals:** description length, name caps ratio

### Modeling
- `ColumnTransformer` pipeline — clean separation of numeric/categorical preprocessing
- 5-fold KFold cross-validation on the winning model
- **SHAP TreeExplainer** for global feature importance — beyond raw importance scores
- Metrics reported in both log space (modeling) and € (interpretability)

### Limitations (stated openly)
- Trained on **host-set asking prices**, not realized booking prices
- 25% of listings dropped due to missing price — selection bias possible
- `neighbourhood_cleansed` has only 4 values in Inside Airbnb's Dublin scrape — relied on `lat/lon` + distance features instead

## 🛠️ Tech decisions worth defending in interview

| Decision | Why |
|---|---|
| `log1p(price)` target | Heavy right-skew; back-transform gives % error interpretation |
| Drop NaN price rows rather than impute | Imputed prices would leak through metrics; ~25% dropped is honest |
| Geodesic distances over neighbourhood encoding | `neighbourhood_cleansed` has 4 values → low signal; lat/lon → 6 informative distances |
| `min_frequency=20` in OneHotEncoder | Avoids overfitting to rare property types |
| Parquet for processed data | 10× faster IO than CSV, preserves dtypes |
| `src/` modules + thin notebooks | Reusable logic; notebooks tell a story, not implement utilities |
